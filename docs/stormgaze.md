# Stormgaze / 暴风眼

> 周围一切在高速运转，你站在最中心，看得最清楚。

---

## 一、项目定位

Stormgaze 是一个面向"控制欲强"的用户的 AI Agent 可视化执行平台。用户下达任务后，Agent 将其拆解为宏观 DAG 拓扑 + 微观动态步骤，实时渲染为可交互的网状执行图。用户可以随时 drill down 查看细节、暂停执行、干预方向、回溯决策。

现有 Agent 产品（ChatGPT、Claude、Manus 等）的交互本质是"线性日志流"——用户只能看着文字滚动，不知道 Agent 走到了哪、接下来要干嘛、还剩多久。开发者工具（LangGraph Studio、Langfuse）虽然有图可视化，但它们面向的是调试场景，不是终端用户。

Stormgaze 要填的就是这个空白：把 Agent 的执行过程从黑箱变成一张用户看得懂、能操控的实时地图。

## 二、核心概念

### 2.1 双层任务拓扑

任务结构分为两层，解决"动态性"和"确定感"之间的矛盾。

**宏观层（Macro Plan）** 是用户下达任务后，Agent 先拆解出的相对稳定的任务 DAG。比如"调研 → 分析 → 对比 → 综合 → 输出"。这一层在任务开始前就确定，给用户一个全局心智模型。宏观节点之间的依赖关系、并行关系一目了然。

**微观层（Micro Steps）** 是每个宏观节点内部的具体执行步骤，由 Agent 在运行时动态生成。比如"调研"节点展开后可能是"搜索论文 → 抓取报告 → 采集竞品 → 交叉验证"。这些步骤会随着执行过程实时出现、更新、完成。

这样设计的好处是：宏观层提供确定感，微观层保留灵活性。用户不会因为看到一张不断剧烈变化的图而焦虑，也不会因为只看到一个进度条而失去掌控感。

### 2.2 可干预执行

用户不是观众，而是指挥官。在任何一个正在执行的节点上，用户可以：

- **暂停**：冻结当前节点的执行，Agent 等待用户决定
- **跳过**：放弃当前微观步骤，推进到下一步
- **调整方向**：对 Agent 下达补充指令，改变后续步骤的策略
- **回溯**：回到已完成节点的某个微观步骤，重新执行
- **插入节点**：在宏观 DAG 中临时新增任务节点

### 2.3 Context 用量计量与 Memory 策略

LLM 的 context window 是有限资源。Stormgaze 需要像管理内存一样管理 context。

**计量机制：** 每个微观步骤产生的输入输出都计入 context 消耗量。UI 上持续展示当前 context 的用量水位（类似内存占用条）。用户可以预设 LLM 的 context 大小（如 8K / 32K / 128K / 200K），系统根据预设值计算剩余空间。

**自动 Memory 处理：** 当 context 用量接近预设阈值（如 80%）时，系统自动触发 memory 压缩：

- 已完成节点的详细执行记录被压缩为摘要（保留关键结论，丢弃中间过程）
- 摘要被标记为"已归档"，用户仍可在 UI 上展开查看原始记录（从本地存储读取，不占用 context）
- 正在执行和待执行的节点保持完整 context

**分级存储：** context window 内只保留"活跃信息"——当前节点的完整记录 + 其他节点的摘要。完整的历史记录持久化在本地（公网版用 IndexedDB，私有版用后端存储），需要回溯时再加载。

## 三、部署架构

### 3.1 公网版（纯前端）

公网版是一个纯静态站点，可部署在 Vercel / Cloudflare Pages / 任何静态托管上。不存在后端服务器。

- **LLM 调用：** 前端直接用用户自己配置的 API 地址、模型名称、Secret Key 调用 LLM。请求从用户浏览器直接发往 LLM 服务商，不经过任何中间服务器。用户的 Key 只存在本地浏览器中。
- **代码执行：** 浏览器内沙箱。前端项目（HTML/CSS/JS/React/Vue）通过 WebContainer 运行；Python 脚本通过 Pyodide (WASM) 运行。沙箱内的代码无法访问用户本机文件系统，安全隔离由浏览器保证。
- **数据存储：** 项目数据、执行历史、Memory 归档全部存储在 IndexedDB 中，不离开用户浏览器。支持导出/导入项目快照。
- **能力边界：** 只能运行 JavaScript/TypeScript 和纯 Python（含 numpy/pandas/matplotlib 等 WASM 兼容包）。无法运行需要系统调用、GPU、数据库、或其他语言运行时的代码。UI 上对此有明确提示，并引导用户切换至私有部署版。

### 3.2 私有部署版（前端 + 后端）

私有部署版通过 Docker Compose 一键启动，运行在用户自己的机器或服务器上。前端 UI 与公网版共用同一套代码。

- **Runtime Server：** 一个轻量后端服务，负责管理沙箱化的代码执行环境。通过容器化隔离（Docker-in-Docker 或 gVisor）确保 Agent 生成的代码不会影响宿主系统。
- **语言支持：** 不受浏览器限制，可运行任意语言——Node.js、Python（含 PyTorch 等重型库）、Go、Rust、Java 等。由 Runtime Server 按需拉取对应的语言镜像。
- **沙箱安全策略：** 每次代码执行在独立容器内完成，限制网络访问范围、文件系统挂载范围、CPU/内存用量、最大执行时间。执行完成后容器销毁，不留残余状态。
- **数据存储：** 项目数据可选择存储在本地文件系统或用户自建的数据库中。

### 3.3 运行时抽象层

公网版和私有版共享同一套 Runtime Interface。上层的 Agent 逻辑和 UI 不需要知道代码到底在哪执行。

Runtime Interface 定义四个能力：

- **createEnvironment(language)** — 创建指定语言的执行环境
- **writeFiles(files)** — 向环境写入文件
- **execute(command)** — 执行命令，流式返回 stdout/stderr
- **preview(port)** — 获取预览地址（Web 项目）

公网版的实现：WebContainerRuntime / PyodideRuntime。私有版的实现：DockerRuntime。切换部署模式只需替换 Runtime 实现，零改动上层代码。

## 四、项目管理

Stormgaze 支持多项目管理。每个项目是一个独立的工作空间。

**项目结构：**

- 项目名称、描述、创建时间
- LLM 配置（API URL、模型名称、Secret Key、Context 大小预设）——每个项目可以绑定不同的模型
- 任务历史：该项目下所有执行过的任务，每个任务保留完整的宏观 DAG + 微观步骤记录 + 产出文件
- 文件空间：Agent 在该项目中产出的所有文件，用户可浏览、下载、删除

**项目数据持久化：**

- 公网版：IndexedDB，支持导出为 .stormgaze 文件（JSON 打包），可跨设备导入
- 私有版：后端持久化，支持多用户协作（远期）

## 五、LLM 配置

Stormgaze 不绑定任何 LLM 服务商。用户在设置页面自行配置：

- **API URL：** 兼容 OpenAI API 格式的任意端点（OpenAI、Anthropic、本地 Ollama、各类代理服务等）
- **Model Name：** 用户手动输入模型标识符
- **Secret Key：** 仅存储在浏览器本地，不上传、不中转
- **Context 大小预设：** 用户根据所选模型设定 context window 上限（如 gpt-4o 设 128K，claude-opus 设 200K，本地小模型设 8K），系统据此计算用量水位和触发 memory 压缩的阈值

支持配置多个 LLM Profile，在项目间或任务间切换使用。

## 六、前端设计原则

以下五条原则贯穿 Stormgaze 全部界面设计，不可违背。

### 6.1 禁止 3D 光泽球体

任何需要表达"节点"、"状态"、"实体"的地方，一律使用发光光晕（glow / bloom）替代 3D 球体。光晕可以通过颜色、强度、脉动频率传达状态信息（完成/执行中/等待），同时保持视觉上的扁平和轻量。不使用任何 3D 渲染、阴影球体、玻璃质感元素。

### 6.2 禁止 Emoji

全平台不出现任何 emoji 字符。所有图标一律使用自行设计的 SVG icon，保持视觉语言的统一性和品牌辨识度。SVG icon 的设计风格应与整体 UI 调性一致——线条感、几何感、克制。

### 6.3 禁止"边框卡片"设计语言

不使用"圆角边框 + 左上角 logo + 大字 title + 小字 description"这种典型的 SaaS 卡片式布局。这是当前 B 端产品最泛滥的设计模式，与 Stormgaze 的气质不符。信息层次通过排版节奏、字重对比、留白、光效来建立，而不是靠边框来框定区域。

### 6.4 明确的交互引导

用户第一次打开 Stormgaze 时，需要知道该做什么、该看哪里、该怎么操作。交互引导不是弹窗教程或蒙层指引，而是通过设计本身传达——视觉焦点引导注意力、动效暗示可交互区域、空状态提供明确的行动入口。任何功能点都不能让用户"不知道这个东西能点"或"不知道点了会怎样"。

### 6.5 拒绝传统 SaaS 设计语言

Stormgaze 的视觉不应该像 Notion、Linear、Figma 或任何一个"好看的 SaaS"。它应该更接近"指挥中心"或"深空监控站"的气质——暗色基调、信息密度高但层次清晰、动态元素有节制但有力、整体散发一种"安静的掌控感"。设计参考方向：航天任务控制台、电影里的战情室、深海探测器的仪表盘。

## 七、技术选型

### 7.1 前端

- 框架：React 18+ (TypeScript)
- 构建：Vite

- 容器管理：Docker Engine API
- 通信协议：WebSocket（前端与 Runtime Server 之间的流式通信）
- 部署方式：Docker Compose 一键启动

## 八、Context 计量细节

### 8.1 计量单位

以 token 为基本单位。前端内置一个轻量 tokenizer（如 tiktoken 的 WASM 版本）进行近似估算。不同模型的 tokenizer 存在差异，Stormgaze 提供"估算模式"（通用 BPE）和"精确模式"（用户指定 tokenizer）两档。

### 8.2 计量对象

每次 LLM 调用时，计量以下内容的 token 数：

- System Prompt（Agent 人设、工具定义等固定开销）
- 宏观 DAG 结构描述
- 已完成节点的摘要（memory 压缩后的版本）
- 当前节点的完整执行记录
- 用户的原始任务描述与中途干预指令
- Agent 的历史输出

### 8.3 水位线与 Memory 触发

三条水位线：

- **安全区（0-60%）：** 正常执行，不做任何处理
- **预警区（60-80%）：** UI 上 context 用量条变为警示色，提示用户 context 即将紧张。此时系统开始预准备 memory 摘要（后台异步生成已完成节点的摘要），但不立刻替换
- **压缩区（80-100%）：** 自动执行 memory 压缩。将已完成节点的完整记录替换为摘要，释放 context 空间。被替换的原始记录保存到本地存储，用户可在 UI 上展开查看但不会被送入 LLM

### 8.4 Memory 摘要策略

摘要由 LLM 自身生成。对每个已完成节点，构造一次专门的摘要调用："将以下执行记录压缩为不超过 N token 的摘要，保留关键结论、数据、决策依据，丢弃中间推理过程。"摘要结果缓存在本地，后续不再重复生成。

## 九、MVP 范围

第一版聚焦核心体验，不做完美主义。

**包含：**

- LLM 配置页面（API URL / Model / Key / Context Size）
- 项目创建与管理（新建、列表、删除、切换）
- 任务输入与宏观 DAG 自动生成
- DAG 实时渲染（节点状态、连线动画、进度指示）
- 微观步骤面板（展开查看当前节点内部执行详情）
- 暂停/继续执行
- Context 用量水位条
- 自动 Memory 压缩
- WebContainer 内代码预览（仅 JS/TS 前端项目）

**不包含（后续迭代）：**

- Pyodide Python 运行时
- 私有部署版
- 节点回溯与重新执行
- DAG 手动编辑（插入/删除/重排节点）
- 多 LLM Profile 切换
- 项目导出/导入
- 多用户协作

## 十、命名体系

产品名：**Stormgaze / 暴风眼**

宏观节点 → **Storm Node / 风暴节点**
微观步骤 → **Pulse / 脉冲**
Context 用量条 → **Pressure Gauge / 气压计**
Memory 压缩 → **Condense / 凝结**
任务执行中状态 → **Eye Open / 眼开**
任务完成状态 → **Eye Closed / 眼闭**
项目 → **Theater / 战场**

所有命名延续"风暴观测"隐喻体系，保持产品语言的一致性和辨识度。
